## 1. 개요 및 비유
**Controller Manager**는 클러스터 내의 다양한 '상태 감시 로봇'들을 관리하는 대장입니다. 각각의 컨트롤러는 특정 자원을 감시하며 바라는 상태를 유지하려고 노력합니다.

💡 **비유하자면 '아파트 관리소의 무인 감시 시스템'과 같습니다.**
* **복제 컨트롤러:** "각 층에 소화기가 3개씩 있는지 확인하고 부족하면 채워넣어."
* **노드 컨트롤러:** "경비원(Node)이 응답이 없으면 사퇴 처리하고 보고해."
* **엔드포인트 컨트롤러:** "식당 메뉴가 바뀌면 안내판(Service/Endpoint)을 즉시 수정해."

## 2. 핵심 설명
* **지속적인 루프:** "현재 상태 확인 $\rightarrow$ 차이점 발견 $\rightarrow$ 수정 작업 수행"이라는 무한 루프를 돌며 클러스터를 안정화합니다.
* **하나의 프로세스:** 논리적으로는 여러 컨트롤러(Node, Job, Endpoint 등)가 존재하지만, 복잡성을 줄이기 위해 `kube-controller-manager`라는 하나의 실행 파일로 묶여서 돌아갑니다.

## 3. YAML 적용 예시 (Controller Manager 실행 옵션)
노드 장애를 얼마나 빨리 감지할지 결정하는 설정 예시입니다.

```yaml
# kube-controller-manager 실행 파라미터 중 일부
- --node-monitor-grace-period=40s    # 40초 동안 응답 없으면 비정상 의심
- --node-monitor-period=5s          # 5초마다 노드 상태 체크
- --pod-eviction-timeout=5m0s       # 노드 다운 시 5분 후 파드를 다른 곳으로 이사시킴
```

## 4. 트러블 슈팅
* **파드가 다른 노드로 바로 이사 가지 않는 경우:**
  * 노드가 죽었을 때 즉시 파드가 살아나지 않는다면 `pod-eviction-timeout` 설정을 확인하세요. 기본값이 5분이라 성격 급한 한국 엔지니어들에게는 장애처럼 느껴질 수 있습니다.
