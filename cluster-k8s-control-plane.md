## 1. 개요 및 비유
**Control Plane(컨트롤 플레인)**은 쿠버네티스의 '뇌' 역할을 하는 컴포넌트들의 집합입니다. 클러스터의 전반적인 결정(스케줄링, 이벤트 응답 등)을 내리고 클러스터의 상태를 관리합니다.

💡 **비유하자면 '회사의 본사(경영진)'와 같습니다.**
실제 현장에서 뛰는 직원(Worker Node)들이 일을 잘할 수 있도록 전략을 짜고, 인력을 배치하고, 문제가 생기면 보고를 받아 결정을 내리는 경영진 전체를 의미합니다.


## 2. 핵심 설명
* **구성 요소:** `kube-apiserver`, `etcd`, `kube-scheduler`, `kube-controller-manager`로 구성됩니다.
* **Desire State 유지:** 사용자가 선언한 '바라는 상태'를 감시하고, 실제 상태가 그와 다를 경우 명령을 내려 두 상태를 일치시킵니다.
* **고가용성 (HA):** 본사가 무너지면 회사 전체가 마비되므로, 보통 3개 이상의 노드에 컨트롤 플레인을 분산 배치하여 장애에 대비합니다.

## 3. YAML 적용 예시 (Pod 스펙 내의 컨트롤 플레인 컴포넌트 확인)
정적 파드(Static Pod) 형태로 실행되는 컨트롤 플레인 컴포넌트들의 매니페스트 위치를 확인하는 것이 주니어 시절엔 중요합니다.

```bash
# 워커 노드가 아닌 마스터 노드 터미널에서 확인
ls /etc/kubernetes/manifests/

# 예시: kube-apiserver.yaml 내용 중 일부

```

## 4. 트러블 슈팅
* **컨트롤 플레인 노드 부족:**
  * 홀수(1, 3, 5...) 단위로 구성하는 것이 원칙입니다. 2대로 구성했다가 1대가 죽으면 `etcd`가 과반수 찬성(Quorum)을 얻지 못해 클러스터 전체가 Read-only 상태가 되거나 멈출 수 있습니다.
